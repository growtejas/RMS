 1. EXECUTIVE SUMMARY
================================================================================

The RBM Resource Fulfillment Module is an enterprise-grade, workflow-driven
platform for managing the complete lifecycle of project requisitions — from
creation by a Project Manager, through multi-level budget and HR approval,
to Talent Acquisition execution, candidate pipeline management, and position
fulfillment.

KEY CHARACTERISTICS:
  - Multi-role enterprise system (Manager, HR, TA, Admin, Owner)
  - Strict state-machine workflow engine with matrix-based governance
  - Dual-level workflow: header (requisition) + item (position) lifecycles
  - Full audit trail with immutable transition logging
  - Optimistic + pessimistic concurrency control
  - Role-based UI rendering with backend-enforced authorization
  - Candidate pipeline with interview tracking
  - Real-time status governance preventing unauthorized state mutations

PRODUCTION METRICS:
  - 20+ database entities, fully normalized
  - 2,400+ lines of workflow engine code
  - 15+ API router modules, 50+ endpoints
  - 4,300+ line TA detail component (feature-rich)
  - Complete candidate pipeline (Sourced → Hired)
  - Full budget approval workflow at item granularity

OPENING SLIDE SUMMARY:
  "A workflow-driven hiring management platform that enforces strict state-
   machine governance across multi-level approvals, role-based candidate
   pipelines, and audited position fulfillment — built on FastAPI, React,
   PostgreSQL, and a custom transition-matrix engine."

 4. WORKFLOW ENGINE ANALYSIS
================================================================================

4.1 WORKFLOW MODEL
-------------------
The system implements a DUAL-LEVEL STATE MACHINE:

  HEADER LEVEL (Requisition):
    Draft → Pending_Budget → Pending_HR → Active → Fulfilled
                 ↓                ↓          ↓
              Rejected         Rejected   Cancelled
                 ↓
               Draft (resubmission via F-004)

  ITEM LEVEL (RequisitionItem):
    Pending → Sourcing → Shortlisted → Interviewing → Offered → Fulfilled
                 ↑           ↓              ↓             ↓
              (re-source) (backward)    (backward)    (backward)
    Any non-terminal → Cancelled

  KEY RELATIONSHIP: Header status drives item editability:
    - Items can only transition when header is Active
    - Header auto-fulfills when all items reach terminal state (SYSTEM-only)
    - Pending_Budget/Pending_HR blocks item modifications

4.2 TRANSITION ENFORCEMENT
----------------------------
  1. Matrix Validation: Every transition checked against HEADER_TRANSITIONS
     or ITEM_TRANSITIONS dictionaries
  2. Role Authorization: HEADER_TRANSITION_AUTHORITY and
     ITEM_TRANSITION_AUTHORITY define per-transition role requirements
  3. System-Only Guards: Certain transitions (Pending→Sourcing,
     Active→Fulfilled) are reserved for SYSTEM role only
  4. Backward Transition Enforcement: Backward transitions require mandatory
     reason (minimum character length)
  5. Status Protection Layer: SQLAlchemy before_update listener blocks ALL
     direct status mutations outside workflow_transition_context() — GC-001

4.3 AUTHORIZATION ENFORCEMENT
-------------------------------
  Header-Level Examples:
    Draft → Pending_Budget     : Manager
    Pending_Budget → Pending_HR: Manager, Admin, HR
    Pending_HR → Active        : HR, Admin
    Active → Fulfilled         : SYSTEM only
    Rejected → Draft           : Manager, Admin (resubmission)

  Item-Level Examples:
    Pending → Sourcing         : SYSTEM only (TA assignment triggers)
    Sourcing → Shortlisted     : TA, Admin
    Interviewing → Offered     : TA, HR, Admin
    Offered → Fulfilled        : HR, TA

4.4 CONCURRENCY HANDLING
--------------------------
  Dual Strategy:
    1. Optimistic Locking: `version` column on Requisition and RequisitionItem.
       Engine validates expected_version before transitions.
    2. Pessimistic Locking: SELECT FOR UPDATE on critical operations
       (_get_locked_item, _get_locked_requisition) prevents phantom reads
       during concurrent transitions.

4.5 TRANSACTION BOUNDARIES
----------------------------
  - Engine methods NEVER commit — caller owns the transaction
  - Audit writes are in-transaction (failure = rollback of transition)
  - Triple audit: WorkflowTransitionAudit + AuditLog + StatusHistory
  - Notifications fire post-commit to avoid side-effect rollback issues

4.6 STRENGTHS
--------------
  + Matrix-driven: Adding new transitions requires only matrix entries
  + GC-001 enforcement: Impossible to bypass engine via direct SQL
  + Comprehensive audit: Three complementary audit tables
  + Extensible hooks: Post-transition hook registry for notifications/metrics
  + Clean exception hierarchy: Each error type maps to specific HTTP codes
  + Version tracking in audit: Before/after versions logged per transition

4.7 RISKS & COMPLEXITY
------------------------
  - Engine file is 2,400+ lines — could benefit from decomposition
  - Legacy engine (V1) still exists (disabled) — technical debt
  - Thread-local workflow context relies on single-threaded request model
  - No distributed locking for multi-instance deployment

 5. ROLE & PERMISSION MODEL
================================================================================

5.1 SYSTEM ROLES
-----------------
  Role      Description                          Primary Capabilities
  --------  -----------------------------------  --------------------------------
  Manager   Project/resource manager              Create requisitions, set budgets,
                                                  submit for approval, track progress
  HR        Human Resources administrator         Approve budgets, approve HR,
                                                  assign TAs, manage employees,
                                                  cancel items (Kill Switch)
  TA        Talent Acquisition specialist          Self-assign items, source
                                                  candidates, manage pipeline,
                                                  conduct interviews, extend offers,
                                                  fulfill positions
  Admin     System administrator                  Full system access, user
                                                  management, override capabilities
  Owner     Business owner                        Top-level dashboard, strategic
                                                  view, full admin capabilities
  SYSTEM    Automated system actor                Auto-transitions (Pending→Sourcing,
                                                  Active→Fulfilled)

5.2 AUTHORIZATION STRATEGY
----------------------------
  - Backend Enforced: Every API endpoint uses Depends(require_any_role(...))
  - Workflow Enforced: Transition matrix defines per-transition role sets
  - Frontend Enforced: ProtectedRoute component checks roles before rendering
  - Item-Level TA Ownership: TA must be assigned to specific item (or header
    fallback) to perform operations on that item
  - Candidate Ownership: TA ownership resolved via item → requisition chain

5.3 SECURITY BOUNDARIES
--------------------------
  - TA can only self-assign (cannot assign other TAs)
  - HR/Admin can assign any TA to any item
  - Direct status mutation blocked at ORM level (GC-001)
  - Budget fields protected: estimated_budget immutable during approval phase
  - Token-based identity: JWT payload contains user_id, username, roles

6.2 DESIGN STRENGTHS
----------------------
  + Normalized: Proper 3NF with clear entity boundaries
  + Constrained: CHECK constraints on status fields, budget non-negativity,
    currency format, fulfilled-must-have-employee (GC-004)
  + Indexed: Status columns, foreign keys, and lookup fields indexed
  + Foreign Keys: RESTRICT deletes prevent orphans on critical relationships,
    CASCADE on dependent relationships (candidates, interviews)
  + Computed Properties: Budget totals computed dynamically (not stored),
    avoiding stale aggregate data
  + Version Columns: Optimistic locking on requisitions and items
  + Triple Audit: Three complementary audit tables for different query patterns

6.3 SCALABILITY CONSIDERATIONS
---------------------------------
  - All primary keys are auto-increment integers (except emp_id: string)
  - Status columns indexed for filtered queries
  - Entity_type + entity_id indexing on audit tables supports polymorphic queries
  - pool_pre_ping=True for connection health checking
  - No materialized views or denormalized query tables (room for optimization)

 7. FEATURE MODULE BREAKDOWN
================================================================================

7.1 REQUISITION CREATION (Manager)
------------------------------------
  Purpose    : Managers create resource requisitions for their projects
  Workflow   : Fill form → Add items → Attach JD → Submit for approval
  Backend    : POST /api/requisitions/ (multipart: form data + JD file)
               Items created inline via items_json field
  UI         : RequisitionWizard.tsx — multi-step form with validation
               Professional modal popup on success/failure
  Data Flow  : Manager fills form → POST creates Draft → Submit transitions
               to Pending_Budget → Budget approval pipeline begins

7.2 BUDGET APPROVAL (HR / Admin)
-----------------------------------
  Purpose    : Two-phase approval: Budget clearance then HR validation
  Workflow   : Pending_Budget → Budget review → Pending_HR → HR approval → Active
  Backend    : PATCH /api/requisitions/{id}/approve-budget
               PATCH /api/requisitions/{id}/approve-release (combined)
               Workflow item-level budget: edit/approve/reject per item
  UI         : HRGatekeeperPanel.tsx — per-item budget editing
               ItemBudgetApprovalPanel.tsx — dashboard pending budget view
               Budget Approved By field required before approval
  Key Rule   : Estimated budget is IMMUTABLE during approval phase;
               HR edits approved_budget only

7.3 HIRING EXECUTION (TA)
---------------------------
  Purpose    : TA manages entire candidate pipeline per position item
  Workflow   : Pending → Sourcing (auto on TA assign) → Shortlisted →
               Interviewing → Offered → Fulfilled
  Backend    : POST /api/requisition-items/{id}/workflow/assign-ta
               POST /api/requisition-items/{id}/workflow/shortlist
               POST /api/requisition-items/{id}/workflow/start-interview
               POST /api/requisition-items/{id}/workflow/make-offer
               POST /api/requisition-items/{id}/workflow/fulfill
  UI         : RequisitionDetail.tsx (TA view) — 4-tab layout:
               Overview | Items | Candidates | Timeline
               Candidate pipeline with stage filters
               Interview scheduling and result tracking
  Key Rule   : TA must be assigned to item (or header fallback) for access

7.4 CANDIDATE PIPELINE
------------------------
  Purpose    : Track applicants through sourcing → hiring stages
  Workflow   : Sourced → Shortlisted → Interviewing → Offered → Hired/Rejected
  Backend    : POST /api/candidates/ (create)
               PATCH /api/candidates/{id}/stage (transition)
               POST /api/interviews/ (schedule)
               PATCH /api/interviews/{id} (result/feedback)
  UI         : CandidateDetailModal.tsx — full candidate profile with
               interview rounds, stage progression, resume management
  Data Flow  : TA adds candidate → Links to item → Schedules interviews →
               Records results → Moves to Offered → Hired triggers item fulfill

7.5 ASSIGNMENT MANAGEMENT
---------------------------
  Purpose    : TA assignment at header and item level
  Header     : PATCH /api/requisitions/{id}/assign-ta — propagates to all
               unassigned non-terminal items via workflow engine
  Item       : POST /api/requisition-items/{id}/workflow/assign-ta
  Reassign   : POST /api/requisitions/{id}/items/{id}/reassign-ta (swap)
               POST /api/requisitions/{id}/bulk-reassign
  Key Rule   : Header assignment now uses item workflow engine for each
               item, ensuring Pending→Sourcing auto-transition

7.6 REPLACEMENT FLOW
----------------------
  Purpose    : Handle position replacements for departed employees
  Mechanism  : Items flagged with replacement_hire=true and replaced_emp_id
  UI         : Replacement badge on item cards, replaced employee reference
  Backend    : RequisitionItem.replacement_hire + replaced_emp_id columns

7.7 PROGRESS TRACKING & TIMELINE
-----------------------------------
  Purpose    : Visual progress and audit history for stakeholders
  UI         : Milestone timeline (raised → budget → HR → assigned → fulfilled)
               Completion progress bar with fractional display
               Auto-closure indicator when all items resolved
  Backend    : Status history, workflow audit, general audit — all queryable
               per entity

7.8 SECURITY & AUTHENTICATION
-------------------------------
  Purpose    : Secure multi-role access to all system features
  Login      : POST /api/auth/login → JWT token → localStorage
  Protection : ProtectedRoute component + require_any_role() dependency
  Session    : Axios interceptor injects Bearer token on every request
               401 response triggers automatic redirect to login

 9. SECURITY ANALYSIS
================================================================================

9.1 AUTHENTICATION FLOW
--------------------------
  1. User submits credentials → POST /api/auth/login
  2. Backend verifies password hash (pbkdf2_sha256)
  3. JWT created with sub=user_id, username, roles, exp
  4. Token stored in localStorage as "authToken"
  5. Axios interceptor attaches Bearer token to every request
  6. Backend security dependency extracts and validates JWT per request

9.2 AUTHORIZATION ENFORCEMENT
-------------------------------
  Layer 1 — Route Protection : ProtectedRoute checks user.roles client-side
  Layer 2 — API Dependency   : require_any_role() middleware on every endpoint
  Layer 3 — Workflow Engine   : Transition authority matrix validates role
  Layer 4 — Ownership Check   : TA ownership verified for candidate/item ops
  Layer 5 — Status Protection : GC-001 blocks unauthorized status mutations

9.3 DATA PROTECTION
---------------------
  - Passwords one-way hashed (pbkdf2_sha256, never stored in plaintext)
  - JWT secret loaded from environment variable (not hardcoded)
  - Database credentials URL-encoded from .env
  - CORS restricted to specific origins
  - Foreign key RESTRICT prevents accidental data deletion

9.4 INPUT VALIDATION
----------------------
  - Pydantic schemas validate all request payloads
  - CHECK constraints at database level (status enums, budget ranges,
    currency format, priority values)
  - Frontend form validation (required fields, min lengths)
  - Budget non-negativity enforced at DB and API levels

9.5 POTENTIAL IMPROVEMENTS
----------------------------
  - Token refresh mechanism not yet implemented
  - Rate limiting on authentication endpoints recommended
  - HTTPS enforcement should be mandated in production
  - localStorage token storage — consider httpOnly cookies for XSS protection
  - CSRF protection not explicitly implemented


================================================================================
 10. CONCURRENCY & CONSISTENCY
================================================================================

10.1 SIMULTANEOUS ACTIONS
----------------------------
  - Optimistic Locking: Requisition.version and RequisitionItem.version
    columns increment on every transition. If expected_version doesn't match,
    ConcurrencyConflictException is raised (HTTP 409).
  - Pessimistic Locking: SELECT FOR UPDATE on critical operations ensures
    no phantom reads during transition validation.

10.2 RACE CONDITIONS
----------------------
  - Dual-lock strategy prevents two TAs from fulfilling the same item
  - Transaction isolation prevents interleaved audit writes
  - workflow_transition_context (thread-local) prevents bypassing
    the engine for status mutations

10.3 CONFLICTING TRANSITIONS
-------------------------------
  - Transition matrix is deterministic: only one valid target per
    (source, action) pair
  - Terminal states have empty transition sets (no escape)
  - Backward transitions require explicit reason (auditability)

10.4 TRANSACTION ISOLATION
----------------------------
  - Engine methods never commit — caller controls transaction boundary
  - Audit writes are in-transaction (atomicity guaranteed)
  - Database-level CHECK constraints provide last-line-of-defense validation

10.5 DATA INTEGRITY
----------------------

 15. PRESENTATION TALKING POINTS

15.1 TWO-MINUTE SYSTEM EXPLANATION
-------------------------------------
"This platform manages the full lifecycle of project hiring requests.
 A Project Manager creates a requisition specifying positions needed.
 The system enforces a multi-stage approval pipeline — first budget
 clearance, then HR validation. Once approved, Talent Acquisition
 specialists are assigned to source candidates, conduct interviews,
 and fulfill positions. Every action goes through a strict workflow
 engine that enforces role-based permissions and logs an immutable
 audit trail. The system tracks both the overall requisition status
 and each individual position independently."

15.2 FIVE-MINUTE ARCHITECTURE EXPLANATION
--------------------------------------------
"The architecture follows a clean four-layer backend design: API routers
 handle HTTP concerns, a service layer implements business logic through
 a custom state-machine workflow engine, a workflow matrix defines all
 valid transitions and role authorizations, and SQLAlchemy models manage
 the PostgreSQL data layer.

 The workflow engine is the heart of the system. It implements a dual-level
 state machine — header-level for the requisition lifecycle (Draft through
 Fulfilled) and item-level for each position (Pending through Fulfilled).
 Every transition is validated against a matrix, checked for role
 authorization, protected by optimistic locking, and logged to three
 complementary audit tables.

 On the frontend, React with TypeScript provides role-specific dashboards
 for each user type. The frontend mirrors the backend's canonical workflow
 types for type safety and delegates all transition decisions to the backend.
 An Axios interceptor handles JWT injection automatically.

 What makes this unique is the governance layer: SQLAlchemy event listeners
 physically prevent any code from mutating status fields outside the workflow
 engine — even a developer mistake in a new endpoint would be blocked at
 the ORM level."

15.3 ANTICIPATED QUESTIONS AND ANSWERS
-----------------------------------------

Q: "How do you prevent someone from bypassing the workflow?"
A: "Three mechanisms: (1) API endpoints use role-based dependency injection,
    (2) the workflow engine validates every transition against the matrix,
    and (3) SQLAlchemy event listeners block any direct status mutation
    outside the workflow context. Even a raw ORM update would be caught."

Q: "What happens if two people try to approve the same requisition?"
A: "We use dual concurrency control: optimistic locking via version columns
    detects conflicts, and SELECT FOR UPDATE provides pessimistic locking
    during critical sections. The second actor receives a 409 Conflict."

Q: "How is the audit trail structured?"
A: "Triple audit: WorkflowTransitionAudit captures detailed transition data
    with version tracking; AuditLog provides general-purpose event logging;
    RequisitionStatusHistory gives a queryable status timeline. All writes
    are in-transaction — if audit fails, the transition rolls back."

Q: "Can the workflow be customized for different organizations?"
A: "Yes. The transition matrix is defined in a single file
    (workflow_matrix.py). Adding or modifying transitions, authorized roles,
    or new states requires only changes to that matrix — no engine
    modification needed."

Q: "What's the candidate pipeline model?"
A: "Candidates are linked to specific requisition items (positions). They
    progress through stages: Sourced → Shortlisted → Interviewing → Offered
    → Hired. Each stage can have multiple interview rounds with structured
    feedback. When a candidate is hired, it triggers the item fulfillment
    workflow."

Q: "How does budget approval work?"
A: "Each requisition item has separate estimated and approved budget fields.
    The estimated budget is set by the manager and becomes immutable once
    submitted. During the approval phase, HR reviews and sets the approved
    budget independently. A 'Budget Approved By' field is required for
    accountability. Header-level budget totals are computed dynamically
    from items — never stored — preventing stale data."
